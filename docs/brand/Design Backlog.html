what's parked

design work we didn't finish — documented for future you

The brand and the bot-surface sketches are done. What's below was intentionally deferred — you said "keen to build it already," which is the right call for a hobby project. But these are real open design questions, not loose ends, and they'll want answering when you start coding the bot.

Each item has a tag: DESIGN = needs me (or you with the brand book); BACKEND = needs code before design makes sense; QUICK = under an hour; DEEP = real work.

01

Hero layout, finalized DESIGNDEEP

The one we have is a round-one sketch. Not to spec.

Pin dimensions, type scale, and spacing. Recompose using the canonical palette and avatar from the brand book — the sketch used pre-lock values. Write the real copy for every state (bonfire lit, lighting the fire…, fire's going out, bonfire's out) using the voice and vocabulary from the brand book, not the placeholders.

Include the invoker attribution line ("lit by @otso") and a consistent footer block.

02

Line layout, finalized DESIGNQUICK

Status is personal + ephemeral + low-key. The sketch half-gets there.

Decide: does Line use an embed color strip, a plain message, or rich text with one emoji? What's the minimum that still feels like Bonfire? This is probably the most-used surface — /valheim status runs a hundred times for every /valheim start. Worth getting small and right.

03

Alert / Card layout DESIGNQUICK

Errors should feel like Bonfire too, not like default Discord.

Unauthorized, not_found, multiple_instances, AWS-unreachable. Each gets an embed with the right tone (warm apologetic for unauthorized; matter-of-fact for not_found; slightly embarrassed for multiple). Pull copy patterns from the brand book's "words we use / words we don't" section.

04

The state matrix DESIGNQUICK

Which layout gets used when — a single canonical table.

Rows: every EC2 state × every command. Cells: which layout, which copy, ephemeral or public, edit-in-place or one-shot. This is the spec your bot implementation reads from. Small artifact, high leverage.

05

Cold-start edit sequence DESIGNBACKENDDEEP

The 90-second wait is the hardest UX moment in the whole bot.

Storyboard the sequence of edits to a single Hero message as the server boots: t=0 "lighting the fire…" → t=30s "still lighting… (0:30)" → t=90s "🔥 bonfire lit · <ip>". Nail the cadence, the copy per tick, and what happens if it fails past 3 minutes.

This requires backend changes (deferred response + follow-up edits via PATCH /webhooks/… with a poll loop somewhere — Lambda won't want to block for 90s). Design and backend are entangled; worth doing together.

06

Stop confirmation + input polish DESIGNBACKENDQUICK

Stopping is destructive; make it a one-tap "are you sure" with a button.

Design the confirm prompt (embed + two buttons: "put it out" / "nevermind"), including timeout-revert behavior. Also: command autocomplete hints — what does Discord show when a user types /valheim ? Both worth ~30 min of design and maybe an hour of Go to wire up.

Nothing here is urgent. Start building — the brand book plus the sketched surfaces are enough for a first pass. When you hit a real question ("what does the message say while it's booting?"), come back here and pick the matching item. That's when the design work will be easy, because you'll know exactly what you need.
